Xara XAR/WEB File Format Notes for sample0
===========================================

Scope and status
----------------
This document describes the byte-level container and the subset of the Xara graphic model implemented in xara.py. The parser was verified against every file under sample0/: 100 files, 97,015 logical records, 282 raw-deflate compression sections, and no unaccounted trailing bytes. Every physical byte in the samples is consumed as one of the following: the 8-byte file identifier, a record header, record payload, raw-deflate bytes, or the uncompressed TAG_ENDCOMPRESSION CRC/length payload.

The SVG renderer is not a complete Xara renderer. It implements the documented core records needed to produce an SVG approximation for every sample, but it does not faithfully implement all Xara effects. Unsupported or approximate render areas are listed below. The parser still validates record framing, decompression CRCs, and tree navigation for those records.

Top-level byte structure
------------------------
All sample files start with this exact 8-byte identifier:

    58 41 52 41 a3 a3 0d 0a    ASCII "XARA" + marker + CR LF

The rest of the file is a stream of records. All normal multi-byte fields are little-endian unless a record explicitly says otherwise.

Record header:

    UINT32 Tag
    UINT32 Size
    BYTE   Payload[Size]

TAG_FILEHEADER is always the first logical record after the identifier. TAG_ENDOFFILE is always the final logical record. TAG_DOWN and TAG_UP records impose the document tree. A TAG_DOWN makes subsequent records children of the previous non-navigation record until its matching TAG_UP.

Compression sections
--------------------
TAG_STARTCOMPRESSION has Size=4. The payload is three version bytes plus a compression type byte; all samples use compression type 0. Despite the historical spec calling this zlib, the sample files use raw DEFLATE streams with no zlib wrapper.

Immediately after TAG_STARTCOMPRESSION payload, raw-deflate data begins. Inflating it yields normal records. The last inflated record is TAG_ENDCOMPRESSION, but only its 8-byte header is inside the compressed stream. Its Size field is 8. The following 8 physical bytes in the outer file are not deflated and contain:

    UINT32 CRC32 of the inflated bytes
    UINT32 number of inflated bytes

xara.py validates both fields. This is how it can account for compressed sections exactly while still presenting a single logical record stream to the renderer.

Core scalar types
-----------------
BYTE: unsigned 8-bit integer.
UINT16/UINT32: little-endian unsigned integers.
INT32/LONG/MILLIPOINT: little-endian signed 32-bit integers.
COORD: two INT32 values, X then Y, in millipoints. The coordinate system has X increasing right and Y increasing upward. SVG output negates Y while serializing geometry, text anchors, image transforms, and paint-server coordinates so SVG text and embedded bitmaps remain upright.
FIXED16: INT32 divided by 65536.
DOUBLE: IEEE-754 little-endian 64-bit float.
COLOURREF/BITMAPREF/REF: signed 32-bit sequence reference. Positive references point to prior logical records. Negative references point to built-in defaults.
STRING: user-visible strings are UTF-16LE in the samples; file-header producer strings are ASCII NUL-terminated.

Implemented rendering records
-----------------------------
The converter renders these records directly or as close SVG equivalents:

- TAG_PATH and TAG_PATH_* variants: absolute verb/coordinate paths.
- TAG_PATH_RELATIVE and TAG_PATH_RELATIVE_* variants: refined paths, with each entry stored as one verb byte plus an interleaved big-endian coordinate. The first coordinate is absolute; subsequent coordinates store previous-current deltas.
- TAG_PATHREF_TRANSFORM: transforms a previously seen path by the documented 2D matrix.
- TAG_DEFINECOMPLEXCOLOUR: uses the fallback RGB triplet at the start of the record.
- TAG_FLATFILL, TAG_FLATFILL_NONE/BLACK/WHITE, TAG_LINECOLOUR, TAG_LINECOLOUR_NONE/BLACK/WHITE.
- TAG_LINEWIDTH, TAG_STARTCAP, TAG_JOINSTYLE, TAG_WINDINGRULE, and default TAG_DASHSTYLE values.
- TAG_FLATTRANSPARENTFILL and TAG_LINETRANSPARENCY as SVG opacity for mix-style transparency.
- TAG_LAYERDETAILS visibility flags are honored; layers whose visible bit is clear, such as MouseOver and hidden photo layers in the website samples, are parsed but not rendered.
- Multi-page web-template files that encode several pages as repeated Page background/MouseOff layer blocks at the same spread coordinates are split into separate SVG files. Later page phase blocks are parsed and rendered independently, with each output restricted to that page's visible layer set.
- TAG_LINEARFILL, TAG_LINEARFILL3POINT, TAG_CIRCULARFILL, TAG_ELLIPTICALFILL, and TAG_CONICALFILL as SVG paint servers. TAG_FILLEFFECT_FADE emits the normal two-stop RGB interpolation. TAG_FILLEFFECT_RAINBOW and TAG_FILLEFFECT_ALTRAINBOW emit sampled HSV hue ramps using the documented shortest/longest hue route; conical fills use the same colour interpolation for their wedge colours. Conical fills are emitted as user-space angular SVG patterns built from wedge polygons because SVG has no native conical gradient in the target profile.
- TAG_SQUAREFILL is mapped to the same three-point radial paint server used for elliptical fills. TAG_FOURCOLFILL is mapped to a transformed SVG pattern that approximates the four-corner colour interpolation with layered linear gradients. TAG_SQUARETRANSPARENTFILL updates fill opacity from its start/end transparency bytes.
- TAG_BITMAPFILL and TAG_CONTONEBITMAPFILL as SVG pattern paint servers. Contone remapping is not applied.
- TAG_DEFINEBITMAP_JPEG, TAG_DEFINEBITMAP_JPEG8BPP, TAG_DEFINEBITMAP_PNG, TAG_NODE_BITMAP, and bitmap fill references as embedded SVG images. TAG_DEFINEBITMAP_PNG 32bpp RGBA alpha is normalized by inverting the stored alpha channel as specified by the XAR format; without this, Xara's internally stored alpha-channel PNGs render in browsers with opaque black transparent regions. TAG_NODE_BITMAP stores its four corners as top-left, top-right, bottom-right, bottom-left, followed by the bitmap reference.
- TAG_ELLIPSE_SIMPLE, TAG_ELLIPSE_COMPLEX, TAG_RECTANGLE_SIMPLE, TAG_RECTANGLE_SIMPLE_ROUNDED, TAG_RECTANGLE_COMPLEX, TAG_RECTANGLE_COMPLEX_ROUNDED, and TAG_REGULAR_SHAPE_PHASE_2. Rectangle quickshapes use the documented bounding-ellipse construction: the stored width/height or major/minor axes describe the ellipse, and the upright rectangle corners are the four equally spaced points on that ellipse, so the actual rectangle side vectors are scaled by 1/sqrt(2). TAG_REGULAR_SHAPE_PHASE_2 flag bit 0x01 is circular, bit 0x02 is stellated, bit 0x04 enables primary curvature, and bit 0x08 enables stellation curvature. Circular regular shapes are rendered as ellipses, not stars; stellation offset is a ratio of one side angle, not degrees. The two embedded edge paths that follow the fixed 75-byte phase-2 header are parsed; straight two-point edges and single-Bezier custom edge templates are rendered by scaling/rotating the edge template to each regular-shape side, matching Xara's InsertShapeEdge construction.
- TAG_POLYGON_COMPLEX and TAG_POLYGON_COMPLEX_ROUNDED_STELLATED approximated as polygon/star paths. Rounded corner geometry is not faithfully reproduced.
- TAG_RECTANGLE_COMPLEX_STELLATED and TAG_POLYGON_COMPLEX_STELLATED are rendered as stellated regular-shape paths using the stored centre, axes, stellation radius ratio, and stellation offset ratio.
- Simple TAG_BLEND records now generate intermediate paths. When adjacent source/destination objects have compatible path command layouts their commands are interpolated directly. Incompatible path layouts fall back to sampled-polyline interpolation so files such as Watch4.xar still render blend interiors instead of only endpoints. TAG_BLENDER stores start/end path indexes; TAG_BLENDERADDITIONAL stores curve/object index metadata and the reverse flag. Blend-on-curve and non-path object conversion are still simplified.
- TAG_SHADOWCONTROLLER/TAG_SHADOW wall, floor, and glow shadows are mapped to SVG filter effects. Wall shadows use the stored X/Y offset, penumbra blur, shadow colour, transparency, and darkness; floor shadows render a transformed shadow-only alpha copy using the recorded floor angle, floor height, offset, penumbra, shadow colour, transparency, and darkness; glow shadows use a zero-offset blur. Feather shadow transforms are not implemented.
- TAG_FLATTRANSPARENTFILL is emitted as SVG fill opacity. TAG_LINEARTRANSPARENTFILL, TAG_CIRCULARTRANSPARENTFILL, TAG_ELLIPTICALTRANSPARENTFILL, TAG_SQUARETRANSPARENTFILL, and TAG_LINEARTRANSPARENTFILL3POINT are emitted as SVG alpha masks with user-space linear or radial gradients, so Xara colour fills and transparency fills remain independent. TAG_FEATHER is emitted as a path-shaped SVG alpha mask blurred with feGaussianBlur using the recorded feather width; this supports feathered bitmap-filled shapes such as vignette.xar. If a filled-and-stroked object has a gradient transparency fill or feather, the converter emits separate fill and stroke paths so the mask affects the fill only. Xara transparency type 2 (stained glass) maps to SVG mix-blend-mode:multiply and type 3 (bleach) maps to mix-blend-mode:screen; the other SVG-addressable blend modes are mapped where CSS has an equivalent. Bitmap/fractal and conical transparency fills are still reduced to a flat average opacity.
- Basic text story/string/font-size/justification/typeface/style records are converted to SVG text. Text is emitted as styled runs, so nested fill color, gradient fill, font family, size, bold, italic, tracking, and baseline changes inside a text story are preserved where SVG has a direct equivalent. TAG_TEXT_STORY_COMPLEX and the complex start/end-left/right variants use their story matrices converted from Xara's Y-up coordinates into SVG's Y-down coordinate space; TAG_TEXT_LINE and TAG_TEXT_LINE_INFO records are used for hard line breaks, recorded line offsets, and recorded line extents. Single-run straight text lines with a recorded width are emitted with SVG textLength/lengthAdjust so browser font metrics do not shrink or expand the line relative to Xara's layout. TAG_TEXT_JUSTIFICATION_FULL is recognized, but SVG output leaves it as start-aligned text because SVG text has no direct automatic full-justification equivalent in this renderer.
- Text stories that contain a single path child and one text line are emitted as SVG textPath content. This handles the text-on-curve construction used by samples such as photo_corner_bottom_right.xar; the path is stored in defs and the text is anchored with the story's justification.
- Text lines whose story has a wrap width are moved to the right of overlapping bitmap-filled objects on the same vertical line, matching the text-repel/floating-photo layout used by the website samples.
- TAG_PREVIEWBITMAP_GIF is parsed and can be used as a fallback only if no visible SVG elements are produced.

Converter output behavior
-------------------------
xara.py takes an input XAR/WEB file and an existing output directory:

    xara.py <inputFile> <outputDirectory>

It renders every discovered page block as a separate SVG in that directory. When the file supplies page names with TAG_4372, the SVG filenames are the sanitized page names, for example index.svg, products.svg, store.svg, news.svg, gallery.svg, contact.svg, blog.svg, and about.svg in the trader_website sample. If no page name is present, the converter uses a generated page name or, for a single unnamed page, the input file stem.

Fail-fast behavior
------------------
xara.py writes page SVGs through temporary files and renames them only after all pages render successfully. It requires the output directory to already exist. It fails on missing magic, malformed record headers, record sizes that exceed the stream, unsupported compression type, raw-deflate failure, compression CRC mismatch, compression length mismatch, malformed path records, unbalanced TAG_DOWN/TAG_UP navigation, or a non-directory output path.

Known incomplete areas and next steps
-------------------------------------
This is not a full Xara renderer. The following sample-present features are parsed but either skipped or approximated:

- Blends: only compatible path-to-path blends are interpolated. Xara's full blend algorithm, object mapping, blend-on-curve, and non-path object conversion are not fully implemented.
- Moulds: TAG_MOULD_PERSPECTIVE records with quadrilateral mould paths are rendered by remapping emitted path, bitmap-corner, and fill-control coordinates through a projective transform from the source bounding box to the mould quadrilateral. TAG_MOULD_ENVELOPE records use a bilinear corner mapping. This handles the perspective mould stacks in Watch4.xar much more faithfully than bbox scaling, but it is still not Xara's full curve-by-curve envelope deformation engine.
- Compound rendering/effects: TAG_COMPOUNDRENDER and higher Xtreme effect records are not rendered as offscreen composited bitmaps.
- Live/locked effects, brush records, set/bar metadata, and newer 43xx/44xx records are not fully specified by the supplied 2007 PDF and are not rendered faithfully. Feather attributes are handled for filled paths and bitmap nodes, but full Xara feather-effect offscreen compositing is still more sophisticated than this SVG mask mapping.
- Bitmap/fractal transparency fills, conical transparency fills, multi-stage transparency ramps, fill profiles, transparency modes without direct SVG/CSS equivalents, and repeat/inverted-repeat tiling modes are simplified.
- Text layout is improved but still simplified; exact shaping, kerning, font substitution, aspect-ratio scaling, exact tab-stop positioning, arbitrary vector-object text repel, and path-fitted text are not reproduced.
- Rounded QuickShape corners are approximated for some polygon/star cases.

To make this complete, the next engineering step would be to obtain the newer Xara record definitions for the 4161-4434 tag range and implement a real rendering model for blends, moulds, transparency compositing, feather/effect bitmaps, and all fill geometries. The parser is structured so those can be added record-by-record without changing the byte accounting layer.
