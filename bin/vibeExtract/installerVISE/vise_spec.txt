Installer VISE Archive Format Specification
============================================
Version: 2026-02-21
Source:   Reverse-engineered from sample files and ScummVM project (elasota, 2022)

Installer VISE was a proprietary macOS (Classic Mac OS) installer creation tool
by MindVision Software, widely used from the mid-1990s through early 2000s.
The archive format stores Mac files (with both data and resource forks) in a
compressed, obfuscated container.

All multi-byte integers are BIG-ENDIAN (Motorola byte order).

================================================================================
FILE STRUCTURE OVERVIEW
================================================================================

A VISE archive consists of:

  1. SVCT Header (44 bytes)
  2. Compressed file data (variable length)
  3. CVCT Catalog (variable length)
     - DVCT entries (directories)
     - FVCT entries (files)
     - PACK entries (install packages)
     - INDN entry (disk name table)

For multi-segment (multi-disk) archives:
  - Segment 1 contains the SVCT header, some/all compressed data, and the CVCT
  - Segments 2+ contain only additional compressed data (no header or catalog)
  - Each segment file begins with its own SVCT header

================================================================================
SVCT HEADER (44 bytes)
================================================================================

Offset  Size  Field                Description
------  ----  -------------------  -------------------------------------------
0x00    4     magic                "SVCT" (0x53564354)
0x04    4     segment              Segment number (1 = first/catalog segment)
0x08    8     unknown              Reserved/unknown fields
0x10    4     version              Archive format version (see below)
0x14    16    unknown              Reserved/unknown fields
0x24    4     cvct_offset          Byte offset of CVCT catalog in this file
0x28    4     unknown              Possibly total archive size or checksum

Known version codes:
  0x00000001  -  VISE format v0.01
  0x80010101  -  VISE format v1.01
  0x80010201  -  VISE format v2.01
  0x80010202  -  VISE format v2.02 (Installer VISE 3.5 Lite)
  0x80010300  -  VISE format v3.00 (Installer VISE 3.6 Lite)
  0x80010301  -  VISE format v3.01
  0x80010302  -  VISE format v3.02
  0x80010303  -  VISE format v3.03
  0x80010304  -  VISE format v3.04
  0x80010305  -  VISE format v3.05
  0x80010306  -  VISE format v3.06
  0x80010307  -  VISE format v3.07
  0x80010308  -  VISE format v3.08
  0x8001030a  -  VISE format v3.10
  0x8001030b  -  VISE format v3.11
  0x8001030c  -  VISE format v3.12
  0x8001030d  -  VISE format v3.13
  0x8001030e  -  VISE format v3.14
  0x80010a50  -  VISE format v10.80
  0x80010b00  -  VISE format v11.00

The version code format is 0x8001XXYY where XX.YY is the version number.
(Exception: v0.01 uses code 0x00000001.)

Catalog format by version:
  v0.01-v3.01:  Sequential parsing, uncompressed DVCT/FVCT entries
  v3.02-v3.07:  Marker-based scanning, uncompressed DVCT/FVCT entries
  v3.08-v3.11:  May be uncompressed (marker scan) OR compressed (same as v3.12+)
  v3.12+:       Compressed catalog blob (byte-swap + raw deflate, NO substitution)
                Decompressed catalog contains standard DVCT/FVCT entries

Archives may be standalone (SVCT at offset 0) or embedded in self-extracting
Mac OS applications (SVCT typically at offset 128).

Archive data layout:
  Most archives: [SVCT header][file data...][CVCT catalog]  (catalog at end)
  Some archives: [SVCT header][CVCT catalog][file data...]  (catalog at start)
  When the catalog is near the start (cvct_offset < ~4096), archive_pos values
  in FVCT entries will be GREATER than cvct_offset. This is the "catalog-first"
  layout seen in some multi-segment installers (e.g., QuickTime Installer v3.08).

================================================================================
CVCT CATALOG HEADER (20 bytes)
================================================================================

Located at the byte offset specified by SVCT.cvct_offset.

Offset  Size  Field                Description
------  ----  -------------------  -------------------------------------------
0x00    4     magic                "CVCT" (0x43564354)
0x04    4     catalog_data_size    For v3.08+: compressed catalog blob size (when compressed)
0x08    4     unknown              Always 1
0x0C    4     unknown              Always 0
0x10    2     num_entries          Total number of catalog entries (DVCT + FVCT)
0x12    2     unknown              Reserved (always 0)

Note: In some archives, the SVCT header's cvct_offset field does not point to
the actual CVCT location. When the CVCT magic is not found at the claimed
offset, search the data for a valid CVCT header (validate by checking that
num_entries is reasonable, e.g. < 5000). If the CVCT is at a different
position, all position_in_archive values in FVCT entries may need to be
adjusted by the same offset difference (shift = actual_cvct - claimed_cvct).

For v0.01-v3.11:
  Immediately after the CVCT header, catalog entries follow sequentially.
  Each entry begins with a 4-byte magic identifier.

For v3.12+:
  After the CVCT header (20 bytes), there is an 80-byte preamble:
    +00..+27: 7 x 4-byte resource type tags:
              "PACK" "BBrd" "CODE" "BDIR" "PROJ" "INDN" "DfIL"
    +28..+79: 13 x uint32 offset values (absolute from SVCT header)
  Then at offset +80, the compressed catalog blob (catalog_data_size bytes):
    1. Byte-swap adjacent pairs (same as file data step 1)
    2. Raw deflate decompress (NO substitution cipher - unlike file data)
    3. The decompressed data contains standard DVCT/FVCT/PACK entries
  Note: Some v3.12+ archives have an UNCOMPRESSED catalog (the blob starts
  with a known marker like FVCT, DVCT, PACK, etc.). In this case, skip
  byte-swap + deflate and use the raw blob data directly.

================================================================================
DVCT ENTRY (Directory)
================================================================================

Magic: "DVCT" (0x44564354)

Offset  Size  Field                Description
------  ----  -------------------  -------------------------------------------
0x00    4     magic                "DVCT"
0x04    78    directory_data       Directory metadata (see below)
 +N     var   extra_bytes          Version-dependent extra bytes (see below)
 +N     var   name                 Directory name (Mac Roman encoding)

Directory data fields (offsets relative to directory_data start):
  +68   2     containing_dir       Parent directory index (1-based, 0=root)
  +76   1     name_length          Length of directory name in bytes

For v10.80/v11.00, name_length is shifted by +2:
  +78   1     name_length          (shifted from +76)

Extra bytes before name (version-dependent):
  v2.01, v2.02:  0 extra bytes
  v3.00, v3.01:  6 extra bytes
  v3.02+:        varies (auto-detect by entry size)

Total entry size: 4 + 78 + extra_bytes + name_length

================================================================================
FVCT ENTRY (File)
================================================================================

Magic: "FVCT" (0x46564354)

This is the primary file descriptor. It describes a single installable file
with its Mac type/creator codes, compressed data sizes, and archive position.

Offset  Size  Field                Description
------  ----  -------------------  -------------------------------------------
0x00    4     magic                "FVCT"
0x04    120   file_data            File metadata (see below)
 +N     var   name                 File name (Mac Roman encoding)

File data fields (offsets relative to file_data start, i.e., byte 4 of entry):

  +0    40    unknown              Various metadata (install flags, etc.)
  +40   4     file_type            Mac OS file type (e.g., "APPL", "TEXT")
  +44   4     file_creator         Mac OS creator code (e.g., "ttxt", "MSWD")
  +48   2     finder_flags         Finder flags (fdFlags from FInfo)
  +50   6     unknown              Finder location/folder fields
  +56   4     creation_date        Creation date (seconds since 1904-01-01)
  +60   4     modification_date    Modification date (seconds since 1904-01-01)
  +64   4     comp_data_size       Compressed data fork size in bytes
  +68   4     uncomp_data_size     Uncompressed data fork size in bytes
  +72   4     comp_rsrc_size       Compressed resource fork size in bytes
  +76   4     uncomp_rsrc_size     Uncompressed resource fork size in bytes
  +80   12    unknown              Additional metadata
  +92   2     containing_dir       Parent directory index (1-based, 0=root)
  +94   2     data_segment         Segment number containing this file's data
  +96   4     position_in_archive  Byte offset of compressed data in segment
  +100  18    unknown              Additional metadata
  +118  1     name_length          Length of file name in bytes
  +119  1     padding              Unused byte

For v10.80/v11.00, all fields after +118 are shifted by +2:
  +120  1     name_length          (shifted from +118)
  +121  1     padding              Unused byte

Total entry size: 4 + 120 + name_length (v0.01-v3.14)
                   4 + 122 + name_length (v10.80, v11.00)

Notes on compressed sizes for shared blobs:
  When multiple FVCT entries share the same position_in_archive value, they
  form a "shared blob" group. In this case:
    - comp_data_size = total compressed size of the shared blob
    - comp_rsrc_size = total UNCOMPRESSED size of all files in the group
    - uncomp_data_size = this file's uncompressed data fork portion
    - uncomp_rsrc_size = this file's uncompressed resource fork portion
  The decompressed blob contains: [all data forks concatenated in catalog
  order] followed by [all resource forks concatenated in catalog order].

  When an FVCT entry has a unique position_in_archive, the standard layout
  applies:
    - Data fork: comp_data_size compressed bytes at position_in_archive
    - Resource fork: comp_rsrc_size compressed bytes at
      position_in_archive + comp_data_size

Later versions (v3.04+) may have additional bytes in the file_data section,
shifting name_length and position_in_archive to higher offsets. The fields
at offsets +40 through +80 (type, creator, compressed sizes) remain stable
across known versions.

================================================================================
PACK ENTRY (Install Package)
================================================================================

Magic: "PACK" (0x5041434b)

Represents an installation package/option in the installer's UI. Structure
is not fully documented. PACK entries are skipped during file extraction.

================================================================================
INDN ENTRY (Disk Name Table)
================================================================================

Magic: "INDN" (0x494e444e)

Located after the last catalog entry (usually at the end of the file).
Contains the names and metadata for each disk/segment in a multi-disk archive.

Offset  Size  Field                Description
------  ----  -------------------  -------------------------------------------
0x00    4     magic                "INDN"
0x04    1     unknown              Flags or version byte
0x05    1     disk_count           Number of disk name entries

For each disk:
  +0    1     name_length          Length of disk name
  +1    var   name                 Disk name (Mac Roman encoding)
  +N    6     disk_info            Disk metadata (segment number, size, etc.)

================================================================================
PHANTOM ENTRIES
================================================================================

Some FVCT entries in the catalog are "phantom" entries — installer action
references (e.g., package set definitions) rather than extractable files. These
can be identified by their characteristic size field values:

  - comp_data_size AND uncomp_data_size are both "phantom sizes"
  - A phantom size is >= 0x8000 AND matches one of:
      a) N * 65536 + M, where 1 <= N <= 8 and 0 <= M <= 8
         (e.g., 0x10000, 0x10001, 0x10005, 0x20001, 0x40001, 0x40002, 0x80001)
      b) An exact power of 2 (e.g., 0x8000, 0x10000, 0x40000)

These values appear to be two packed 16-bit fields misinterpreted as one 32-bit
value. Phantom entries should be filtered out during extraction as they do not
contain decompressible file data.

================================================================================
COMPRESSION AND OBFUSCATION
================================================================================

VISE uses a three-step compression scheme (reverse-engineered by the ScummVM
project from the installer's "Dcmp" resource):

  1. BYTE-PAIR SWAP
     Adjacent bytes are swapped in pairs:
       for i = 1, 3, 5, ..., len-1:
         swap(data[i], data[i-1])
     (Odd-length data: last byte remains in place)

  2. SUBSTITUTION CIPHER
     Each byte is replaced using a fixed 256-byte lookup table:
       for i = 0 to len-1:
         data[i] = TABLE[data[i]]

     The deobfuscation table (hex):
       6A B7 36 EC 15 D9 C8 73  E8 38 9A DF 21 25 D0 CC
       FD DC 16 D7 E3 43 05 C5  8F 48 DA F2 3F 10 23 6C
       77 7C F9 A0 A3 E9 ED 46  8B D8 AC 54 CE 2D 19 5E
       6D 7D 87 5D FA 5B 9B E0  C7 EE 9F 52 A9 B9 0A D1
       FE 78 76 4A 3D 44 5A 96  90 1F 26 9D 58 1B 8E 57
       59 C3 0B 6B FC 1D E6 A2  7F 92 4F 40 B4 06 72 4D
       F4 34 AA D2 49 AD EF 22  1A B5 BA BF 29 68 89 93
       3E 32 04 F5 DE E1 6F FB  67 E4 7E 08 AF F0 AB 41
       82 EA 50 0F 2A C6 35 B3  A8 CA E5 4C 45 8A 97 AE
       D6 66 27 53 C9 1C 3C 03  99 C1 09 2E 69 37 8D 2F
       60 C2 A6 18 4E 7A B8 CF  A7 3A 17 D5 9E F1 84 51
       0D A4 64 C4 1E B1 30 98  BB 79 01 F6 62 0E B2 63
       91 CB FF 80 71 E7 D4 00  DB 75 2C BD 39 33 94 BC
       8C 3B B6 20 85 24 88 2B  70 83 6E 7B 9C BE 14 47
       65 4B 56 81 F8 12 11 28  EB 55 74 A1 31 F7 B0 13
       86 DD 5F 42 D3 02 61 95  0C 5C A5 CD C0 07 E2 F3

  3. RAW DEFLATE (RFC 1951)
     The result of steps 1+2 is a headerless deflate stream (no zlib or
     gzip header). Decompress using raw inflate (e.g., zlib with
     wbits=-MAX_WBITS).

     Known quirk: VISE's deflate implementation aligns stored blocks to
     2-byte boundaries instead of the standard 1-byte boundary. This
     rarely affects files in practice (stored blocks are uncommon in
     compressible data), but may cause decompression failures for certain
     files. ScummVM notes this as a known TODO.

     Known quirk: Some small files (URL bookmarks, etc.) have a 1-byte
     prefix (typically 0x01 after deobfuscation) before a deflate stored
     block with BFINAL=0. The stored block correctly contains the file data,
     but trailing bytes after the block are invalid. Solution: limit the
     decompressor's output buffer to the expected uncompressed size, which
     stops the decompressor before it encounters the trailing garbage. If
     standard decompression fails, retry after skipping the first byte with
     the output limited to expected_size.

     Known quirk: Some deflate streams produce MORE data than the expected
     uncompressed size (common with shared blob members when tested
     individually, or due to trailing stored blocks). The decompressor's
     output should be limited to exactly the expected uncompressed size.

Decompression procedure:
  1. Read comp_size bytes of compressed data from the archive
  2. Swap byte pairs: for i in range(1, len, 2): swap [i] and [i-1]
  3. Apply substitution table to every byte
  4. Inflate as raw deflate to get uncomp_size bytes

================================================================================
FILE DATA LAYOUT IN ARCHIVE
================================================================================

Compressed file data is stored between the SVCT header (offset 44) and the
CVCT catalog. The layout depends on whether files are individually stored or
packed into shared blobs.

INDIVIDUAL FILES:
  position_in_archive points to the start of the file's compressed data.
  Layout: [data_fork_compressed | resource_fork_compressed]
  - Data fork: comp_data_size bytes → decompress to uncomp_data_size bytes
  - Resource fork: comp_rsrc_size bytes → decompress to uncomp_rsrc_size bytes
  - Resource fork immediately follows data fork (no padding)
  - The next file's data immediately follows (no padding)

SHARED BLOBS (multiple files at same position_in_archive):
  A single compressed blob contains data for multiple files.
  - Compressed blob: comp_data_size bytes at position_in_archive
  - Decompresses to comp_rsrc_size bytes total (note: comp_rsrc_size here
    means the total uncompressed size, NOT a compressed resource size)
  - Decompressed layout: [file1_data | file2_data | ... | fileN_data |
    file1_rsrc | file2_rsrc | ... | fileN_rsrc]
  - Files are ordered by their position in the catalog
  - Each file's uncomp_data_size bytes of data fork come first (all files)
  - Then each file's uncomp_rsrc_size bytes of resource fork (all files)

MULTI-SEGMENT ARCHIVES:
  - Each file's data_segment field (FVCT +94) indicates which segment
    contains its compressed data
  - position_in_archive is relative to the start of that segment file
  - Segment 1 always contains the catalog; other segments contain only data
  - All segment files have their own SVCT header
  - data_segment=0 typically means "data not in this archive" (CD/external)
    or "alternate version." When seg=0 files share a position_in_archive
    with seg=1 files in a shared blob group, the seg=0 files should be
    excluded from the group to avoid corrupting the shared blob protocol.

FILES WITH NO DATA PRESENT:
  Some FVCT entries have uncomp_data_size > 0 but comp_data_size == 0 (or
  similarly for resource forks). These files have catalog metadata but no
  actual compressed data in the archive. They typically represent files that
  would be installed from CD-ROM or another external source. These entries
  should be skipped during extraction.

================================================================================
MAC FILE ATTRIBUTES
================================================================================

Classic Mac OS files have two forks:
  - Data fork: the primary data stream (e.g., document content, executable code)
  - Resource fork: structured resource data (icons, dialogs, strings, etc.)

Both forks are compressed and stored independently in the archive.

File type and creator codes are 4-byte identifiers:
  - Type: identifies the file format (e.g., "APPL"=application, "TEXT"=text)
  - Creator: identifies the creating application (e.g., "ttxt"=TeachText)

File names use Mac Roman encoding (a superset of ASCII with characters for
accented letters, symbols, and Japanese katakana).

================================================================================
VERSION DIFFERENCES
================================================================================

v2.01 (0x80010201):
  - DVCT: 4 + 78 bytes + name (no extra bytes)
  - FVCT: 4 + 120 bytes + name
  - Standard field offsets

v2.02 (0x80010202) "Installer VISE 3.5 Lite":
  - Same structure as v2.01
  - Identical to ScummVM's supported format

v3.00 (0x80010300) "Installer VISE 3.6 Lite":
  - DVCT: 4 + 78 bytes + 6 extra bytes + name
  - FVCT: 4 + 120 bytes + name (same as v2.x)
  - Identical to ScummVM's supported format

v3.01 (0x80010301):
  - Same structure as v3.00

v3.02-v3.03:
  - DVCT extra bytes may differ
  - FVCT mostly compatible with standard 120-byte layout
  - Minor field variations in unused/unknown fields

v3.04:
  - FVCT file_data may have additional bytes (~16 extra)
  - DVCT extra bytes: ~14 bytes
  - Core fields (type, creator, sizes at +64-+80) remain stable

v3.06-v3.07:
  - FVCT file_data significantly larger (~62 extra bytes)
  - Variable-length entries; use marker scanning to find boundaries
  - Core fields (type, creator, sizes at +64-+80, archive_pos at +96,
    name_length at +118) remain at standard offsets

v3.08-v3.11:
  - Same marker-based catalog structure as v3.02-v3.07
  - v3.11 can store file data AFTER the CVCT catalog (allow post-catalog
    archive_pos values)
  - Field offsets remain at standard positions

v3.12-v3.14:
  - Catalog is now compressed (see CVCT structure above)
  - Uses byte-swap ONLY + raw deflate (NO substitution cipher)
  - Decompressed catalog contains standard DVCT/FVCT/PACK entries
  - File data still uses full deobfuscation (swap + substitution + deflate)
  - Variable-length entries; use marker scanning on decompressed data
  - Additional marker types in catalog: BBrd, CODE, BDIR, PROJ, DfIL
  - File data may be stored after the catalog (allow post-catalog positions)

v10.80 (0x80010a50):
  - Same compressed catalog structure as v3.12+
  - FVCT name_length shifted to +120 (instead of standard +118)
  - DVCT name_length shifted to +78 (instead of standard +76)
  - Entry sizes are 2 bytes larger than v3.12 equivalents
  - All other field offsets (type, creator, sizes, archive_pos) unchanged
  - Known samples are CD-based installers where most file data resides on
    separate media (not in the archive file)

v11.00 (0x80010b00):
  - Same compressed catalog structure as v3.12+
  - FVCT name_length shifted to +120 (same as v10.80)
  - DVCT name_length shifted to +78 (same as v10.80)
  - Shared blob format is heavily used (multiple files per archive_pos)
  - File data stored after the catalog; uses standard deobfuscation

================================================================================
VALIDATION (sample11 corpus, 1378 archives)
================================================================================

Tested against a corpus of 1378 VISE archives spanning v0.01 through v3.14,
sourced from various Classic Mac OS software distributions.

Overall fork success rate: 97.1% (193,014 ok, 5,680 fail)

Archive status breakdown:
  PERFECT (all forks decompress):       850  (61.7%)
  GOOD (>90% forks ok):                 267  (19.4%)
  PARTIAL (50-90% forks ok):            158  (11.5%)
  ERROR (can't parse archive):           59  ( 4.3%)
  POOR (<50% forks ok):                   0  ( 0.0%)

Non-extractable archives (59 ERROR):
  - 30x "No extractable files": 28 are Netscape multi-segment stubs (valid
    SVCT/CVCT but zero FVCT entries, all data on other segments); 2 are
    uninstaller archives (e.g., "Remove Office 98") where all FVCT entries are
    phantom entries (installer action references, not extractable files)
  - 24x "No SVCT header": MacBinary-wrapped files that are either truncated
    (128-byte header only) or non-VISE applications
  - 5x "CVCT not found": Truncated Acrobat Reader archives where the catalog
    extends past end of file

Remaining failure categories (5,680 fork failures):
  - ~4,400 deflate errors: Genuine data corruption. 98.3% of these cannot be
    decompressed by any alternative method (raw, swap-only, skip-2, etc.).
    Likely caused by bit rot, incomplete downloads, or CD-ROM read errors.
  - ~900 multi-segment: Files whose data resides on disk segments 2-5 that
    are not present in the archive file being tested.
  - ~110 no-data: FVCT entries with uncomp_size > 0 but comp_size = 0.
    These are catalog entries for files stored on external media (CD-ROM).
  - ~35 size mismatch (under): Deflate stream produces fewer bytes than the
    expected uncompressed size. Likely corrupt or misaligned data.

Version distribution of tested archives:
  v0.01: 4, v2.01: 54, v2.02: 158, v2.05: 1, v3.00: 176, v3.01: 56,
  v3.02: 27, v3.03: 195, v3.04: 105, v3.05: 5, v3.06: 33, v3.07: 181,
  v3.08: 153, v3.10: 51, v3.11: 58, v3.12: 73, v3.13: 15, v3.14: 9
