EPLYBNDS File Format Specification
====================================
Custom binary 3D mesh format (.ply extension, NOT Stanford PLY)
Reverse-engineered from 250 sample files
Vibe coded by Claude


1. FILE FORMAT OVERVIEW
=======================

EPLYBNDS is a binary 3D mesh format that stores geometry with
configurable per-vertex attributes. Despite using the .ply file
extension, it is NOT related to the Stanford PLY format.

All multi-byte values are LITTLE-ENDIAN (x86 byte order).
All floats are IEEE 754 float32 (4 bytes each).
All indices are uint16 (2 bytes each, max observed index: 64271).

File sizes range from 160 bytes to 1.2 MB (250 sample files).


2. FILE STRUCTURE
=================

Files are sequential with no gaps between sections (except
inter-MESH bone remap tables in some skinned files):

  [HEADER]                      32 bytes, always present
  [SKIN]                        Optional, skinned meshes only (3 files)
  [MESH] × N                    1-37 sections per file
  [VERT]                        Exactly 1 section
  [INDX]                        Exactly 1 section
  [ADJA]                        Optional adjacency data (23 files)
  [MROR]                        Optional mirror marker (4 files)


3. HEADER (32 bytes)
====================

  Offset  Size  Type     Description
  0       8     ASCII    Magic: "EPLYBNDS"
  8       4     f32 LE   Bounding box min X
  12      4     f32 LE   Bounding box min Y
  16      4     f32 LE   Bounding box min Z
  20      4     f32 LE   Bounding box max X
  24      4     f32 LE   Bounding box max Y
  28      4     f32 LE   Bounding box max Z

  The bounding box encompasses all vertices in the file.


4. SKIN SECTION (optional, 3 files)
=====================================

Present only in skinned mesh files. Immediately follows the header,
before any MESH sections.

  Offset  Size  Type     Description
  0       4     ASCII    Tag: "SKIN"
  4       4     u32 LE   Bone count (N)
  8+      var   bytes    N bone entries, each:
                           u8(name_length) + ASCII(name)

  Bone entries are packed sequentially with no padding.
  Example: bone_count=25, names like "Bip01", "Bip01 Spine", etc.


5. MESH SECTION (1-37 per file)
================================

Each MESH section defines a sub-mesh that references a contiguous
range of triangles from the shared INDX buffer.

5a. Base Layout (21 + name_len bytes)
--------------------------------------
  Offset  Size  Type     Description
  0       4     ASCII    Tag: "MESH"
  4       4     u32 LE   Format flags (see 5c)
  8       4     u32 LE   Triangle start (in triangle units)
  12      4     u32 LE   Triangle count
  16      1     u8       Material property byte 0 (see 5d)
  17      1     u8       Material property byte 1 (see 5d)
  18      2     bytes    Reserved (always 0x00 0x00)
  [20     4     bytes    Mesh RGBA color — ONLY if mat_props[1] ∈ {0x06, 0x07}]
  var     1     u8       Material name length (N)
  var     N     ASCII    Material name (e.g., "material01.mtl")

  Triangle range in the INDX buffer:
    index_start = tri_start × 3
    index_count = tri_count × 3
  Ranges are sequential non-overlapping partitions.

5b. Inter-MESH Bone Remap Tables
----------------------------------
In skinned files with multiple MESH sections, a bone remap table
may appear between consecutive MESH sections:

  u8(count) + count × u8(bone_index)

  This maps local bone indices for the sub-mesh to the global SKIN
  bone array. Observed in 3 files:
  - skin.ply: 26 bytes (25 bones + count byte)
  - trackl.ply, trackr.ply: 3 bytes each (2 bones + count byte)

  The parser scans forward to the next known tag to skip these.

5c. Format Flags (u32 at offset 4)
------------------------------------
Bit flags encoding the per-vertex attribute layout:

  Bit      Value   Meaning
  ----     -----   -------
  bit 6    0x0040  Has vertex color (4B RGBA between normal and UV)
  bit 3    0x0008  Has tangent vector (16B tangent+w after UV/UV2)
  bit 9    0x0200  Has UV2/lightmap (8B second UV set after first UV)
  bit 12   0x1000  Skinned mesh (bone weight+index between pos and normal)

  Other bits observed but not fully decoded:
  - bit 1  (0x0002): always set in observed files
  - bit 4  (0x0010): always set in non-skinned files
  - bit 8  (0x0100): always set in non-skinned files

  Common flag combinations and resulting strides:
    0x0112  →  stride 32  (pos + normal + UV)
    0x0152  →  stride 36  (pos + normal + color + UV)
    0x011A  →  stride 48  (pos + normal + UV + tangent)
    0x015A  →  stride 52  (pos + normal + color + UV + tangent)
    0x0212  →  stride 40  (pos + normal + UV + UV2)
    0x0252  →  stride 44  (pos + normal + color + UV + UV2)
    0x1118  →  stride 40  (pos + bone_weight + bone_index + normal + UV)

  NOTE: The stride in the VERT header is the AUTHORITATIVE source for
  vertex layout. Use stride, not flags, to determine the byte layout.

5d. Material Properties (bytes 16-19)
---------------------------------------
  Byte 0: Render flags. Most common values:
    0x04 (364 meshes): standard opaque
    0x05 (25 meshes):  possibly alpha-tested
    0x06 (1 mesh):     unknown variant
    0x14 (3 meshes):   possibly double-sided or special blend
    0x44 (5 meshes):   possibly environment-mapped
    0x45 (2 meshes):   variant of 0x44
    Exact bit meanings unknown; does not affect geometry.

  Byte 1: Header extension flag + material type.
    0x00 (32 meshes):  no color, no extension
    0x04 (369 meshes): standard, no extension
    0x05 (17 meshes):  variant, no extension
    0x06 (22 meshes):  EXTENDED — 4 extra RGBA bytes follow
    0x07 (36 meshes):  EXTENDED — 4 extra RGBA bytes follow
    0x0C (3 meshes):   variant, no extension

    When byte 1 ∈ {0x06, 0x07}: 4 bytes of inline RGBA mesh color
    appear at offset 20, before name_len. These provide a per-mesh
    diffuse color (R, G, B, A as u8 values 0-255).

  Bytes 2-3: Always 0x00 0x00 in all 482 observed MESH sections.


6. VERT SECTION (exactly 1 per file)
======================================

Shared vertex buffer used by all MESH sections.

  Offset  Size  Type     Description
  0       4     ASCII    Tag: "VERT"
  4       4     u32 LE   Vertex count (N)
  8       2     u16 LE   Stride (bytes per vertex: 32, 36, 40, 44, 48, or 52)
  10      2     u16 LE   Vertex format flags
  12      var   bytes    N × stride bytes of vertex data

  Total section size: 12 + vertex_count × stride

6a. Vertex Format Flags (u16 at offset 10)
--------------------------------------------
  Observed values and counts:
    7  (210 files): most common, meaning unknown
    1  (17 files):  possibly "basic" format
    0  (23 files):  possibly "no flags"

  Does not affect geometry parsing — stride is authoritative.

6b. Vertex Layouts by Stride
------------------------------

  Stride 32 bytes (123 files, 49.2%) — Basic
    Offset  Size  Type       Description
    0       12    3 × f32    Position (X, Y, Z)
    12      12    3 × f32    Normal (X, Y, Z) — unit length
    24      8     2 × f32    UV texture coordinates (U, V)

  Stride 36 bytes (40 files, 16.0%) — Vertex Color
    0       12    3 × f32    Position
    12      12    3 × f32    Normal
    24      4     4 × u8     Color (R, G, B, A) — 0-255 per channel
    28      8     2 × f32    UV coordinates

  Stride 40 bytes non-skinned (32 files, 12.8%) — Dual UV
    0       12    3 × f32    Position
    12      12    3 × f32    Normal
    24      8     2 × f32    UV coordinates (set 1 — diffuse)
    32      8     2 × f32    UV coordinates (set 2 — lightmap)

  Stride 40 bytes skinned (3 files, 1.2%) — Bone Weights
    0       12    3 × f32    Position
    12      4     f32        Bone weight (0.0–1.0)
    16      4     u32 LE     Bone index (into SKIN bone array)
    20      12    3 × f32    Normal
    32      8     2 × f32    UV coordinates

    Detected by: stride == 40 AND format_flags & 0x1000

  Stride 44 bytes (2 files, 0.8%) — Color + Dual UV
    0       12    3 × f32    Position
    12      12    3 × f32    Normal
    24      4     4 × u8     Color (RGBA)
    28      8     2 × f32    UV coordinates (set 1)
    36      8     2 × f32    UV coordinates (set 2)

  Stride 48 bytes (42 files, 16.8%) — Tangent Vectors
    0       12    3 × f32    Position
    12      12    3 × f32    Normal
    24      8     2 × f32    UV coordinates
    32      12    3 × f32    Tangent vector (X, Y, Z)
    44      4     f32        Tangent W (handedness, typically ±1.0)

  Stride 52 bytes (11 files, 4.4%) — Color + Tangent
    0       12    3 × f32    Position
    12      12    3 × f32    Normal
    24      4     4 × u8     Color (RGBA)
    28      8     2 × f32    UV coordinates
    36      12    3 × f32    Tangent vector
    48      4     f32        Tangent W

  All normals are unit length (magnitude ≈ 1.0).
  In multi-mesh files with mixed format flags, the vertex buffer
  uses the LARGEST needed stride. Unused attribute slots are zeroed.


7. INDX SECTION (exactly 1 per file)
======================================

Shared triangle index buffer. All indices are uint16.

  Offset  Size  Type     Description
  0       4     ASCII    Tag: "INDX"
  4       4     u32 LE   Index count (N) — always a multiple of 3
  8       var   bytes    N × u16 LE index values

  Total section size: 8 + index_count × 2

  Each group of 3 consecutive indices defines one triangle.
  Total triangles = index_count / 3.

  MESH sections reference ranges in triangle units:
    mesh.tri_start to mesh.tri_start + mesh.tri_count
  Multiply by 3 for index offsets into the buffer.


8. OPTIONAL TRAILING SECTIONS
==============================

8a. ADJA — Triangle Adjacency (23 files)
------------------------------------------
  Offset  Size  Type     Description
  0       4     ASCII    Tag: "ADJA"
  4       4     u32 LE   Entry count (N)
  8       var   bytes    N × u16 LE adjacency values

  Stores per-triangle adjacency for shadow volume generation.
  Each triangle has 3 adjacency entries (indices of adjacent triangles
  sharing each edge). Entry count = total_triangles × 3.
  Value 0xFFFF indicates no adjacent triangle (boundary edge).

8b. MROR — Mirror Marker (4 files)
------------------------------------
  Offset  Size  Type     Description
  0       4     ASCII    Tag: "MROR"
  4       4     bytes    Mirror data (exact format: u32 LE flags)

  Indicates the mesh geometry should be mirrored (for symmetric
  vehicle models — left/right track, wheel, etc.).
  Observed in track/wheel files from vehicle models.


9. COORDINATE SYSTEM
====================

EPLYBNDS uses a Y-UP, right-handed coordinate system consistent
with glTF. No coordinate transformation is needed.

Face winding: triangles are stored in CCW order when viewed from
outside. No winding reversal needed for glTF output.


10. CONVERSION NOTES
=====================

10a. Material Color Priority
-----------------------------
For determining the diffuse color of each mesh:
  1. If MESH has extended header (mat_props[1] ∈ {0x06, 0x07}):
     use the inline RGBA mesh color
  2. If vertices have per-vertex color (stride 36, 44, or 52):
     average the vertex colors across the mesh's triangles
  3. Otherwise: default gray (0.5, 0.5, 0.5)

Material names reference external .mtl files that are not included
in the sample set. The names follow patterns like:
  "material01.mtl", "../common/material_#7725.mtl"

10b. Multi-Mesh Files
----------------------
Each MESH section becomes a separate glTF mesh node.
Triangle ranges partition the shared index buffer — all meshes
reference the same vertex buffer but different index ranges.
Materials can differ per mesh.

10c. Skinned Meshes
--------------------
3 files contain SKIN sections with bone data:
  - Per-vertex bone weight (f32) and bone index (u32)
  - Bone name array in SKIN section
  - Inter-MESH bone remap tables
  - Currently converted as static geometry only (no glTF skinning)

10d. Per-Vertex Normals
------------------------
All vertex layouts include per-vertex normals at a fixed offset
(bytes 12-23 for non-skinned, bytes 20-31 for skinned). These are
always unit-length and used directly for smooth shading in glTF.

10e. Tangent Vectors
---------------------
Stride-48 and stride-52 layouts include tangent vectors with a
W component for handedness. These could be used for normal mapping
in glTF but are not currently exported (no textures available).


11. FORMAT STATISTICS
=====================

250 sample files across 32 subdirectories:
  - 250/250 parse and convert successfully (0 errors)
  - 482 total MESH sections
  - 4 to 28,338 vertices per file
  - 2 to 14,556 triangles per file
  - Max observed vertex index: 64,271 (within u16 range)

Vertex stride distribution:
  32B: 123 files (49.2%) — basic pos+normal+UV
  48B:  42 files (16.8%) — with tangent vectors
  36B:  40 files (16.0%) — with vertex colors
  40B:  35 files (14.0%) — 32 dual-UV + 3 skinned
  52B:  11 files (4.4%)  — color + tangent
  44B:   2 files (0.8%)  — color + dual-UV

Optional sections:
  SKIN: 3 files (1.2%)
  ADJA: 23 files (9.2%)
  MROR: 4 files (1.6%)

Extended MESH headers (inline RGBA color):
  mat_props[1]=0x06: 22 meshes across 12 files
  mat_props[1]=0x07: 36 meshes across 17 files

Bone remap tables between MESH sections:
  3 files (skin.ply, trackl.ply, trackr.ply)
