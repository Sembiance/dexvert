================================================================================
                    ORIC DSK Disk Image File Format Specification
================================================================================

Version: 1.0
Date: 2026-03-30

This document describes the ORIC DSK disk image file format used for Oric
Microdisc floppy disk images. There are two container variants (ORICDISK and
MFM_DISK) and three known filesystem types (Sedoric, OricDOS, FTDOS).


================================================================================
1. OVERALL FILE STRUCTURE
================================================================================

  +-------------------+
  | Header (256 bytes)|   <-- Always present, one sector
  +-------------------+
  | Track 0 data      |
  +-------------------+
  | Track 1 data      |
  +-------------------+
  | ...               |
  +-------------------+
  | Track N-1 data    |
  +-------------------+

Total tracks stored = sides * tracks_per_side.

Track ordering for double-sided disks:
  - Raw indices 0 to (tracks_per_side - 1): Side 0, tracks 0..N-1
  - Raw indices tracks_per_side to (2 * tracks_per_side - 1): Side 1, tracks 0..N-1


================================================================================
2. FILE HEADER (256 bytes)
================================================================================

The first 256 bytes of the file form the header sector. All multi-byte integer
fields are little-endian.

  Offset  Size     Field         Description
  ------  -------  -----------   -------------------------------------------
  0x00    8 bytes  Signature     "ORICDISK" (plain) or "MFM_DISK" (MFM-encoded)
  0x08    4 bytes  Sides         Number of disk sides (1 or 2), uint32 LE
  0x0C    4 bytes  Tracks        Tracks per side, uint32 LE
  0x10    4 bytes  Geometry      ORICDISK: sectors per track (uint32 LE)
                                 MFM_DISK: geometry ID (typically 1)
  0x14    236 bytes Padding      Usually zero-filled; may contain tool artifacts
                                 (e.g., leftover PC directory data, offset tables)

File size formula:
  ORICDISK: 256 + (sides * tracks_per_side * sectors_per_track * 256)
  MFM_DISK: 256 + (sides * tracks_per_side * bytes_per_track)
            where bytes_per_track = (file_size - 256) / (sides * tracks_per_side)
            Typical values: 6400 (standard), 3200 (rare, e.g., RANDOS.DSK)


================================================================================
3. ORICDISK (PLAIN) TRACK FORMAT
================================================================================

When the signature is "ORICDISK", track data is stored as raw sector data with
no encoding overhead. Each track consists of sectors_per_track consecutive
sectors of 256 bytes each.

  Track data = sectors_per_track * 256 bytes

Sectors are stored sequentially within each track:
  Sector 1 data (256 bytes) | Sector 2 data (256 bytes) | ... | Sector N data

No gap bytes, sync marks, address marks, CRC bytes, or other MFM overhead is
present. The data is a direct dump of the logical sector contents.


================================================================================
4. MFM_DISK (MFM-ENCODED) TRACK FORMAT
================================================================================

When the signature is "MFM_DISK", each track is stored as a fixed-size block
(typically 6400 bytes) containing a complete MFM-encoded track image. The track
data mimics the byte stream that would be read from a physical floppy disk,
including all gaps, sync patterns, and markers.

4.1 Track Layout Overview
-------------------------

  +------------------+
  | Track preamble   |  ~56-108 bytes of gap and index mark
  +------------------+
  | Sector 1         |  ~346-365 bytes per sector
  +------------------+
  | Sector 2         |
  +------------------+
  | ...              |
  +------------------+
  | Sector N         |
  +------------------+
  | Trailing padding |  Remaining bytes filled with 0x4E
  +------------------+

Total = bytes_per_track (usually 6400 = 0x1900)

4.2 Track Preamble
------------------

The track preamble precedes the first sector and consists of:

  Bytes          Value    Description
  -----------    ------   ----------------------------------
  ~40 bytes      0x4E     Gap 4a (post-index gap)
  12 bytes       0x00     Sync zeros
  3 bytes        0xC2     Index address mark sync (special clock)
  1 byte         0xFC     Index address mark
  ~40 bytes      0x4E     Gap 1 (pre-sector gap)
  12 bytes       0x00     Sync zeros (lead into first sector)

Note: Some disk images omit the index mark (C2 C2 C2 FC) and begin directly
with gap bytes followed by sync zeros. The preamble size varies; the parser
should scan for the first A1 A1 A1 FE pattern.

4.3 Sector Format
-----------------

Each sector consists of an address field and a data field separated by a gap:

  === ADDRESS FIELD ===
  Bytes          Value    Description
  -----------    ------   ----------------------------------
  3 bytes        0xA1     Sync marks (missing clock pattern in MFM)
  1 byte         0xFE     ID Address Mark
  1 byte         (var)    Track number (physical track on this side)
  1 byte         (var)    Side number (0 or 1)
  1 byte         (var)    Sector number (1-based)
  1 byte         (var)    Size code: sector size = 128 << code
                          Code 1 = 256 bytes (standard)
                          Code 2 = 512 bytes
  2 bytes        0xF7F7   CRC placeholder (FDC write-CRC command byte)
                          In real hardware, 0xF7 triggers CRC generation.
                          In disk images, this field is always 0xF7 0xF7.

  === GAP 2 (inter-field gap) ===
  ~22 bytes      0x4E     Gap fill (may also contain 0x22 bytes)
  12 bytes       0x00     Sync zeros

  === DATA FIELD ===
  3 bytes        0xA1     Sync marks
  1 byte         0xFB     Data Address Mark
  256 bytes      (var)    Sector data (or 128 << size_code bytes)
  2 bytes        0xF7F7   CRC placeholder

  === GAP 3 (inter-sector gap) ===
  ~46 bytes      0x4E     Gap fill between sectors

4.4 Sector Spacing and Gap Sizes
---------------------------------

The gap sizes vary depending on the number of sectors per track:

  Sectors/Track   Gap 1 (Track Lead)   Gap 2   Gap 3
  -------------   ------------------   -----   -----
  15-17           ~72 bytes            ~34     ~50
  18              ~12 bytes            ~34     ~46
  19              ~36 bytes            ~34     ~20

Typical sector spacing (address mark to next address mark): ~358 bytes for
standard 17-sector tracks.

4.5 Trailing Padding
--------------------

After the last sector's Gap 3, the remaining bytes in the track block are
filled with 0x4E to reach the fixed track size (6400 bytes).

4.6 Sector Numbering
--------------------

Within each MFM track, sectors are numbered starting at 1 (one-based). The
sector number in the address mark's ID field identifies the logical sector.
Sectors may appear in any order within the track (sector interleaving/skew),
though most images store them sequentially (1, 2, 3, ..., N).

4.7 Number of Sectors Per Track
-------------------------------

The number of sectors per track is NOT stored in the MFM_DISK header. It must
be determined by parsing the MFM data and counting address marks. Common values:
  - 16 sectors per track (e.g., OricDOS disks)
  - 17 sectors per track (most common, standard Sedoric)

Different tracks on the same disk may have different sector counts.

4.8 Variable Track Size
-----------------------

While 6400 bytes per track is standard, the actual bytes_per_track is computed
from the file:
  bytes_per_track = (file_size - 256) / (sides * tracks_per_side)

Values observed: 6400 (standard), 3200 (rare). When bytes_per_track is 3200,
a single logical track's sectors may span two raw track blocks in the file.


================================================================================
5. DOUBLE-SIDED TRACK ENCODING
================================================================================

For double-sided disks, the filesystem uses encoded track numbers to reference
sectors on side 1.

5.1 Sedoric / OricDOS Encoding
------------------------------

  Track reference     Decoded meaning
  ---------------     ----------------------------------------
  0x00 - 0x7F         Side 0, track = value
  0x80 - 0xFF         Side 1, track = (value & 0x7F)

  Example: Track reference 0x94 = Side 1, track 20 (0x94 & 0x7F = 0x14 = 20)

  Raw file index: side 0 track T -> raw index T
                  side 1 track T -> raw index tracks_per_side + T

5.2 FTDOS Encoding
------------------

FTDOS uses contiguous track numbering without the high-bit encoding:

  Track reference     Decoded meaning
  ---------------     ----------------------------------------
  0 to tps-1          Side 0, track = value
  tps to 2*tps-1      Side 1, track = (value - tracks_per_side)

  Example with 41 tracks/side: Track 67 = Side 1, track 26


================================================================================
6. FILESYSTEM DETECTION
================================================================================

The filesystem type is determined by probing known sector locations for
identification strings:

  Probe   Track  Sector  Offset  Size  Looking for
  -----   -----  ------  ------  ----  -----------
  DET0    0      1       0x40    8     "SEDORIC" or "Oric DOS"
  DET1    4      17      0x40    8     "SEDORIC" or "Oric DOS"
  DET2    2      11      0xC8    8     "SEDORIC" or "Oric DOS"
  DET3    0      2       0x22    8     "SEDORIC" or "Oric DOS"

If no match is found, additional checks:
  - Sedoric bitmap signature: Track 20, Sector 2, byte 0 == 0xFF -> Sedoric
  - FTDOS directory marker: Track 20, Sector 2, byte 6 == 0x55 or 0x4C,
    followed by ASCII filename with embedded '.' -> FTDOS

Recognized filesystem types:
  - SEDORIC: Oric Sedoric DOS (most common)
  - ORICDOS: Oric DOS V1.x
  - FTDOS:   FT-DOS (rare)
  - UNKNOWN: Unrecognized filesystem


================================================================================
7. SEDORIC FILESYSTEM
================================================================================

Sedoric organizes disk metadata on Track 20:
  - Sector 1: System sector
  - Sector 2: Bitmap (sector allocation map), may extend to Sector 3
  - Sector 4: First directory sector (linked list)

7.1 System Sector (Track 20, Sector 1)
---------------------------------------

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    4 bytes  Drive table      Default: 0xD0 0xD0 0xD0 0xD0
  0x04    1 byte   Keyboard type    Default: 0x00
  0x05    4 bytes  RENUM params     Default: 100, 0, 10, 0
  0x09    1 byte   Color marker     If 0x1B (ESC): color info follows at 0x0A
  0x0A    1 byte   Paper color      Only if 0x09 == 0x1B; stored as (color + 64)
  0x0B    1 byte   ESC marker       0x1B
  0x0C    1 byte   Pen color        Stored as (color + 64)
  0x0D    17 bytes Disk name        If colors present (17 bytes)
  --- OR (if byte 0x09 != 0x1B): ---
  0x09    21 bytes Disk name        If no colors (21 bytes)
  --- Then: ---
  0x1E    60 bytes Init string      Auto-executed command on boot

7.2 Bitmap Sector (Track 20, Sector 2; optionally Sector 3)
------------------------------------------------------------

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    1 byte   Signature        Always 0xFF
  0x01    1 byte   Reserved         Always 0x00
  0x02    2 bytes  Free sectors     uint16 LE
  0x04    2 bytes  File count       uint16 LE
  0x06    1 byte   Track count      Tracks per side
  0x07    1 byte   Sectors/track    Sectors per track (e.g., 17)
  0x08    1 byte   Dir sectors      Number of directory sectors
  0x09    1 byte   Sided tracks     Track count OR'd with 0x80 if double-sided
  0x0A    6 bytes  Reserved         Zeros
  0x10    240 bytes Bitmap data     1 bit per sector (1=free, 0=allocated)

The bitmap is a linear bit array covering all sectors across all tracks.
Within each byte, bit 0 represents the lowest-numbered sector. Sector
numbering in the bitmap is 0-based (disk sector 1 = bitmap index 0).

If the bitmap data exceeds one sector (large disks), it continues in Sector 3
with the same header format.

7.3 Directory (Track 20, Sector 4, linked list)
-------------------------------------------------

Each directory sector is structured as:

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    1 byte   Next track       Track of next directory sector (0 = last)
  0x01    1 byte   Next sector      Sector of next directory sector
  0x02    14 bytes Header/padding   Reserved

Starting at offset 0x10, directory entries are 16 bytes each (up to 15 per
sector):

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    9 bytes  Filename         Space-padded (0x20)
  0x09    3 bytes  Extension        Space-padded (0x20)
  0x0C    1 byte   Descriptor track Track of file descriptor sector
  0x0D    1 byte   Descriptor sec   Sector of file descriptor sector
  0x0E    1 byte   Total sectors    Data sectors + descriptor sectors
  0x0F    1 byte   Flags            Bit 6 (0x40) = protected/locked

A zero byte at position 0x00 of an entry signals the end of directory entries
in the current sector. The linked list (bytes 0-1 of each sector) continues
to the next directory sector until next_track == 0.

7.4 File Descriptor (pointed to by directory entry)
----------------------------------------------------

The descriptor sector contains file metadata and a sector map:

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    1 byte   Next desc track  Track of continuation descriptor (0 = none)
  0x01    1 byte   Next desc sector Sector of continuation descriptor
  0x02    1 byte   Signature        Always 0xFF
  0x03    1 byte   Flags            Bit 0: executable
                                    Bit 6 (0x40): block type
                                    Bit 7 (0x80): non-block / data type
  0x04    2 bytes  Start address    Load address, uint16 LE
  0x06    2 bytes  End address      Last byte address + 1, uint16 LE
  0x08    2 bytes  Exec address     Execution address, uint16 LE
  0x0A    2 bytes  Data sectors     Number of data sectors, uint16 LE
  0x0C    varies   Sector map       Pairs of (track, sector) bytes

The sector map at offset 0x0C lists pairs of (track_byte, sector_byte) for
each data sector. Up to 122 pairs fit in the first descriptor sector
((256 - 12) / 2 = 122).

If more pairs are needed, the descriptor continues via the linked list at
bytes 0-1. Continuation descriptor sectors store pairs starting at offset
0x02 (up to 127 pairs per continuation sector).

File data size = end_address - start_address (may differ from
data_sectors * 256).

Each (track, sector) pair uses the encoded track numbering from Section 5.1.


================================================================================
8. ORICDOS FILESYSTEM
================================================================================

OricDOS uses a different layout from Sedoric.

8.1 System Sector (Track 0, Sector 1)
--------------------------------------

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x12    1 byte   Dir start sector First directory sector number
  0x13    1 byte   Dir start track  First directory track number
  0x14    2 bytes  Free sectors     uint16 LE
  0x16    2 bytes  Busy sectors     uint16 LE
  0x18    21 bytes Disk name        Space-padded

Note: In OricDOS, (sector, track) byte order may be used in some fields
(reversed from Sedoric's (track, sector) convention).

8.2 Directory Entries
---------------------

Each directory sector has a linked-list header:
  - Byte 0: Next track (0 = last)
  - Byte 1: Next sector
  - Byte 2: Reserved / data byte count marker

Directory entries start at offset 3 and are 16 bytes each:

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    6 bytes  Filename         Space-padded
  0x06    3 bytes  Extension        Space-padded
  0x09    2 bytes  Sector count     uint16 LE
  0x0B    1 byte   First sector     Sector number of first data sector
  0x0C    1 byte   First track      Track number of first data sector
  0x0D    1 byte   Last sector      Sector number of last data sector
  0x0E    1 byte   Last track       Track number of last data sector
  0x0F    1 byte   Flags            File attributes

8.3 OricDOS Sector Chain
-------------------------

OricDOS files use a linked-list structure where each data sector contains a
pointer to the next:

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    1 byte   Next track       0 = last sector in chain
  0x01    1 byte   Next sector      0 = last sector in chain
  0x02    1 byte   Data count       If 0xFF: header sector (see below)
                                    Otherwise: number of data bytes in sector
  0x03    253 bytes Data             Sector data (up to data_count bytes valid)

If the first sector of a file has byte 2 == 0xFF, it contains a file header:

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    1 byte   Next track       Next sector link
  0x01    1 byte   Next sector      Next sector link
  0x02    1 byte   Marker           0xFF (indicates header present)
  0x03    1 byte   Reserved
  0x04    2 bytes  Start address    Load address, uint16 LE
  0x06    2 bytes  End address      End address, uint16 LE
  0x08    2 bytes  Exec address     Execution address, uint16 LE
  0x0A    1 byte   Data count       Bytes of data in this sector
  0x0B    varies   Data             File data (data_count bytes)

Subsequent sectors in the chain always have the 3-byte header (next_track,
next_sector, data_count) followed by data_count bytes of file data.


================================================================================
9. FTDOS FILESYSTEM
================================================================================

FTDOS (FT-DOS) is a less common Oric DOS. It uses Track 20 for system data
but with a different layout from Sedoric.

9.1 Bitmap Sector (Track 20, Sector 1)
---------------------------------------

The bitmap uses 3 bytes per track (24 bits, of which 17 are used for 17
sectors per track). Each bit represents one sector: 1 = free, 0 = allocated.
Bit 0 of the first byte = sector 1, bit 1 = sector 2, etc.

The last 8 bytes (offset 0xF8) contain the disk name (8 bytes, space-padded).

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    N*3 bytes Bitmap          3 bytes per track, N = total tracks
  0xF8    8 bytes  Disk name        Space-padded ASCII

9.2 Directory (Track 20, Sector 2, may chain)
----------------------------------------------

Each directory sector begins with a 6-byte header:

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    1 byte   Next track       Next directory sector track (0 = last)
  0x01    1 byte   Next sector      Next directory sector (0 = last)
  0x02    1 byte   Entry count      Number of entries (or disk geometry info)
  0x03    1 byte   Sectors/track    Sectors per track (e.g., 17)
  0x04    2 bytes  Reserved         Usually 0x00 0x00

Starting at offset 6, directory entries are 18 bytes each (up to 13 per
sector):

  Offset  Size     Field            Description
  ------  -------  ---------------  ------------------------------------------
  0x00    1 byte   Status           0x55 ('U') = unlocked/used
                                    0x4C ('L') = locked/protected
                                    0x00 = empty/deleted
  0x01    8 bytes  Filename         Space-padded ASCII
  0x09    1 byte   Dot separator    Always 0x2E ('.')
  0x0A    3 bytes  Extension        ASCII
  0x0D    1 byte   File type        0x53 ('S') = sequential
  0x0E    2 bytes  Sector count     Total file sectors, uint16 LE
  0x10    1 byte   First track      Track of first file sector (contiguous numbering)
  0x11    1 byte   First sector     Sector of first file sector

FTDOS uses contiguous track numbering (see Section 5.2). File data sectors
are allocated using the bitmap; the data layout within sectors is raw file
content with no per-sector headers or chain pointers.


================================================================================
10. SECTOR ADDRESSING CONVENTIONS
================================================================================

- Sector numbers are 1-based in all filesystem structures. Sector 1 is the
  first sector on a track. When indexing into arrays, subtract 1.

- Track numbers are 0-based. Track 0 is the first track.

- For double-sided disks, the raw file stores all side 0 tracks first, then
  all side 1 tracks. The filesystem references side 1 tracks using encoded
  track numbers (Section 5).

================================================================================
12. NOTES AND QUIRKS
================================================================================

12.1 CRC Fields
  The 2-byte CRC fields in MFM sectors always contain 0xF7 0xF7. This is
  because 0xF7 is the WD1770/1793 FDC command to "write CRC." In a disk
  image, these bytes are stored literally rather than as computed CRC values.

12.2 Header Padding
  The 236 bytes of header padding (offsets 0x14-0xFF) are usually zero but
  may contain arbitrary data from the tool that created the image. Examples
  include MS-DOS directory fragments, track offset tables, or incrementing
  counter values. Only the first 20 bytes (signature + geometry) are
  meaningful.

12.3 Sector Interleaving
  While most disk images store sectors in sequential order (1, 2, ..., N),
  some may use sector interleaving/skew where sectors appear in non-
  sequential order within the MFM track data. The sector number in each
  address mark identifies the logical sector regardless of physical position.

12.4 Index Address Mark
  The C2 C2 C2 FC index address mark at the start of MFM tracks is present
  in most images but may be absent in some. Parsers should not require it.

12.5 Variable Sectors Per Track
  Different tracks on the same disk may contain different numbers of sectors.
  Each track must be parsed independently by scanning for address marks.

12.6 Sedoric Descriptor Flags
  The flags byte (offset 0x03) in a Sedoric descriptor:
    Bit 0 (0x01): File is executable (has an auto-run address)
    Bit 6 (0x40): Block-type file (loaded as a raw memory block)
    Bit 7 (0x80): Data-type file (non-block, sequential access)

12.7 Empty Sedoric Disks
  A freshly formatted Sedoric disk may have no "SEDORIC" identification
  string at the standard probe locations but will still have a valid bitmap
  at Track 20, Sector 2 (starting with 0xFF 0x00). The disk name may appear
  in the system sector (Track 20, Sector 1) at offset 0x09 or 0x0D.
